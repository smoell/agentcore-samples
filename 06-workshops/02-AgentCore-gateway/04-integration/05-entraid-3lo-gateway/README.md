# AgentCore MCP Gateway with EntraID Authentication

Automated setup for an AgentCore MCP Gateway with EntraID inbound authentication and outbound 3LO (three-legged OAuth) for user-delegated access to a downstream API. Includes a browser-based auth onboarding SPA for pre-authorizing access.

## Prerequisites

- AWS CLI v2 configured with credentials for the target account
- Azure CLI (`az`): `brew install azure-cli` (macOS) or [install guide](https://learn.microsoft.com/en-us/cli/azure/install-azure-cli)
- Node.js 18+ and npm
- CDK CLI: `npm install -g aws-cdk`
- `jq` installed
- Access to a Microsoft Entra ID tenant (CIAM/External ID or standard)

## Quick Start (Automated)

The setup script automates everything: EntraID app registrations, OAuth credential provider, CDK deployment, redirect URI wiring, OpenAPI spec configuration, and generates an env file for subsequent redeployments.

### 1. Log in to both clouds

```bash
# Azure — use your EntraID admin account
az login --tenant <your-tenant-id> --allow-no-subscriptions

# AWS — ensure credentials are configured
export CDK_DEFAULT_ACCOUNT=<your-aws-account-id>
export CDK_DEFAULT_REGION=<your-region>
```

> `--allow-no-subscriptions` is needed for CIAM-only tenants that don't have an Azure subscription.

### 2. Run the setup script

For a CIAM (External ID) tenant:
```bash
cd 05-entraid-3lo-gateway
./scripts/setup.sh \
  --tenant-id <your-tenant-id> \
  --tenant-type ciam \
  --ciam-domain <your-ciam-domain> \
  --region <aws-region> \
  --stack-name <stack-name>
```

For a standard EntraID tenant:
```bash
cd 05-entraid-3lo-gateway
./scripts/setup.sh \
  --tenant-id <your-tenant-id> \
  --tenant-type standard \
  --region <aws-region> \
  --stack-name <stack-name>
```

The script will:
1. Create App A (SPA, public client) and App B (Web, confidential client) in EntraID via Graph API
2. Expose `gateway.access` and `weather.read` scopes
3. Create the OAuth credential provider in AgentCore
4. Register the credential provider callback URL in App B
5. Deploy the CDK stack
6. Register API Gateway redirect URIs in App A
7. Update the workload identity return URLs
8. Update the OpenAPI spec with the real API endpoint and redeploy
9. Generate `.env.<StackName>` for use with `redeploy-cdk.sh`

### 3. Test

Open the auth onboarding URL printed at the end of setup:
```
https://<api-endpoint>/auth
```

## Redeploying During Development

After the initial setup, use `redeploy-cdk.sh` for iterating on Lambda code, CDK stack changes, or OpenAPI spec updates. It reads the `.env.<StackName>` file generated by setup and runs `cdk deploy` with all the right context variables.

```bash
# Auto-detects the env file if there's only one
./scripts/redeploy-cdk.sh

# Or specify explicitly
./scripts/redeploy-cdk.sh .env.MyStack
```

No need to re-run the full setup script for code changes.

## Multiple Deployments

To deploy a second independent instance (e.g., for testing), add `--suffix`:

```bash
./scripts/setup.sh \
  --tenant-id <tenant-id> \
  --tenant-type ciam \
  --ciam-domain <domain> \
  --region eu-west-1 \
  --stack-name MySecondStack \
  --suffix v2
```

The suffix ensures unique resource names (IAM roles, API Gateway, Gateway, credential provider). The OIDC provider is automatically reused if one already exists for the same tenant, and the new App A client ID is added to its audience list.

## Tenant Types

The solution supports both EntraID tenant types:

| Type | Authority Host | When to use |
|------|---------------|-------------|
| `ciam` | `<domain>.ciamlogin.com` | External ID / CIAM tenants |
| `standard` | `login.microsoftonline.com` | Regular EntraID (workforce) tenants |

For CIAM tenants, you must provide `--ciam-domain` (the subdomain before `.ciamlogin.com`).

The key difference: CIAM tenants use `ciamlogin.com` for discovery/token endpoints. Using the wrong host causes `authorizationCode must not be null` errors because AgentCore tries to exchange the authorization code at the wrong token endpoint.

## Testing the Auth Flow

### Auth Onboarding (browser)

1. Open `https://<api-endpoint>/auth` in an incognito window
2. Sign in with Microsoft (EntraID)
3. The SPA calls `tools/call` with `_meta.rawElicitation` — shows "Authorization needed" if not yet authorized
4. Click "Authorize" → EntraID consent page → accept
5. Callback page calls `CompleteResourceTokenAuth` via SigV4 (browser-side)
6. Shows "Authorization Successful"

### VS Code MCP

Add to `.vscode/mcp.json`:
```json
{
  "servers": {
    "agentcore-weather-entraid": {
      "type": "http",
      "url": "https://<api-endpoint>/mcp"
    }
  }
}
```

When a user calls a tool before authorizing, the response interceptor rewrites the elicitation into a friendly message pointing them to the auth onboarding SPA. After authorizing via the SPA, tool calls work without elicitation.

---

## Response Interceptor

The Gateway has a response interceptor Lambda that handles elicitation responses (-32042):

- For VS Code: rewrites the elicitation into a friendly "visit the auth onboarding app" message, since VS Code can't handle the 3LO consent redirect
- For the auth onboarding SPA: passes the raw elicitation through so the SPA can extract the authorization URL and drive the consent flow

The SPA signals its intent by including `_meta: { rawElicitation: true }` in the JSON-RPC request. The interceptor checks this flag and skips the rewrite.

The interceptor also handles Gateway quirks:
- `body: null` on 202 notification responses → returns empty dict
- `body` as JSON string → parses back to dict before returning

## What the Setup Script Creates

### In EntraID

| Resource | Purpose |
|----------|---------|
| App A (SPA, public client) | Shared identity for VS Code + auth onboarding SPA. Exposes `gateway.access` scope. |
| App B (Web, confidential client) | Downstream resource server. Exposes `weather.read` scope. Client secret stored in AgentCore. |

### In AWS

| Resource | Purpose |
|----------|---------|
| API Gateway HTTP API | Public endpoint for proxy, weather API, and auth SPA |
| Proxy Lambda | OAuth intermediary, MCP proxy, serves SPA HTML |
| Weather API Lambda | Mock weather data endpoint |
| Elicitation Interceptor Lambda | Rewrites elicitation for VS Code, passes through for SPA |
| AgentCore Gateway | MCP protocol handling, custom JWT auth, tool routing |
| Gateway Target (weather-api) | OpenAPI target with 3LO credential provider |
| OAuth Credential Provider | Stores App B client credentials, handles token exchange |
| IAM OIDC Provider | Enables STS federation with EntraID JWTs |
| IAM Role (auth-onboarding-web-role) | Browser temp credentials for AgentCore API calls |

### Generated Files

| File | Purpose |
|------|---------|
| `.env.<StackName>` | CDK context variables for `redeploy-cdk.sh` |
| `openapi/weather-api-<StackName>.json` | Deployment-specific OpenAPI spec |

## Troubleshooting

### `authorizationCode failed to satisfy constraint: Member must not be null`

The credential provider is using the wrong discovery URL. CIAM tenants require `ciamlogin.com`, not `login.microsoftonline.com`. Recreate with `--tenant-type ciam`.

### `Invalid Bearer token` from the Gateway

The Gateway authorizer uses `allowedAudience` only (not `allowedClients`). EntraID v2.0 tokens use `azp` for the client ID, not `client_id`. AgentCore validates `allowedClients` against `client_id` which doesn't exist in EntraID tokens.

### `Incorrect token audience` on callback page

The IAM OIDC provider's audience list doesn't include this deployment's App A client ID. The setup script handles this automatically for parallel deployments, but if you created the OIDC provider manually, add the client ID: `aws iam add-client-id-to-open-id-connect-provider`.

### `InterceptorException - Received invalid response from interceptor`

The interceptor returned a response the Gateway couldn't parse. Common causes: `body` was `null` (must be `{}` for empty responses), or `body` was a JSON string instead of a dict.

### `AADSTS9010010` during token exchange

The proxy Lambda strips the `resource` parameter from token requests. EntraID v2.0 uses scopes, not resources. VS Code sends `resource` per RFC 9728 but EntraID rejects it.

### `AADSTS9002327: Tokens issued for the 'Single-Page Application' client-type may only be redeemed via cross-origin requests`

The proxy Lambda adds an `Origin` header when calling EntraID's token endpoint. Required for SPA (public client) token redemption.

### `tools/list` works but `tools/call` triggers elicitation

Expected. `tools/list` doesn't require the downstream token. Only `tools/call` forces the Gateway to fetch the weather token, triggering elicitation if missing.

### Callback page shows "Missing session data"

The JWT wasn't found in `sessionStorage`. Start the flow from `/auth` again.

### `EntityAlreadyExistsException` for OIDC provider during CDK deploy

The IAM OIDC provider is unique per issuer URL per account. If deploying multiple stacks for the same tenant, the setup script automatically detects and reuses the existing provider.

### VS Code MCP server connection throws auth errors after redeployments

Accumulated inbound auth credentials from previous sessions can cause stale-state errors. Open the command palette (`F1` or `⇧⌘P`) and run "Remove Dynamic Authentication Providers" to clear them, then reconnect.
